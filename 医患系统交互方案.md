# 医患系统交互方案

## 一、整体架构说明

根据你提供的架构图，医生端和患者端需要在**挂号**和**病历**两个核心业务模块进行数据交互。

## 二、挂号交互（患者端 → 医生端）

### 2.1 业务流程
```
患者端线上挂号 → 调用医生端接口 → 医生端挂号系统接收 → 医生接诊
```

### 2.2 现有接口（无需新增代码）

**接口地址：** `POST /register-order/add`

**请求参数：**
```json
{
  "doctorId": 1,           // 医生ID
  "scheduleId": 10,        // 排班ID
  "priority": 0            // 0普通 1急诊
}
```

**请求头：**
```
token: <患者端JWT Token>
```

**响应示例：**
```json
{
  "code": 200,
  "msg": "操作成功",
  "data": 12345  // 返回挂号单ID
}
```

**实现位置：**
- Controller: `RegisterOrderController.java:34-51`
- Service: `RegisterOrderService.addRegisterOrder()`

### 2.3 患者端调用示例

```javascript
// 患者端前端调用
fetch('http://doctor-api.hospital.com/register-order/add', {
  method: 'POST',
  headers: {
    'Content-Type': 'application/json',
    'token': localStorage.getItem('patient_token')
  },
  body: JSON.stringify({
    doctorId: 1,
    scheduleId: 10,
    priority: 0
  })
})
```

### 3.3 数据完整性说明

新增的 `/medical-record/patient-detail/{recordId}` 接口包含：

✅ **病历基础信息**：主诉、现病史、诊断结果  
✅ **AI辅助信息**：AI诊断、AI检查建议、AI用药建议、医生确认状态  
✅ **检查报告**：CT/X光等影像报告，包含**图片URL**（掩码图和原始图）、AI分析、医生确认结论  
✅ **检验报告**：血常规、血脂等化验结果，包含检测值、参考范围、异常标志  
✅ **处方用药**：药品名称、用法用量、AI审方意见  
✅ **时间信息**：各项记录的创建时间

## 四、实现细节

### 4.1 新增文件清单

| 文件 | 说明 |
|------|------|
| `vo/PatientMedicalRecordDetailVO.java` | 患者病历详情VO（包含3个内嵌VO类） |
| `service/MedicalRecordService.java` | 新增接口方法声明 |
| `service/serviceimpl/MedicalRecordServiceImpl.java` | 实现数据组装逻辑 |
| `controller/MedicalRecordController.java` | 新增患者端接口 |

### 4.2 数据关联逻辑

```
MedicalRecord (病历)
  ↓ registerOrderId
RegisterOrder (挂号单)
  ↓ id
MedicalOrder (医嘱)
  ↓ id
CheckOrder (检查检验医嘱)
  ↓ id
CheckReport (检查报告) / LabReport (检验报告)

MedicalRecord
  ↓ registerOrderId
Prescription (处方) ← 通过 RegisterOrderId 关联
```

### 4.3 图片URL说明

检查报告中的图片字段：
- **imageUrl**: 掩码图片（已处理的可视化图片，适合前端直接展示）
- **ctUrl**: 原始CT文件（DICOM格式，需专业查看器）

患者端可以：
1. 直接展示 `imageUrl`（用户友好）
2. 提供下载 `ctUrl` 的入口（专业用途）

## 五、患者端集成指南

### 5.1 挂号功能集成

```javascript
// 1. 患者选择医生和排班
const bookAppointment = async (doctorId, scheduleId) => {
  const response = await fetch('http://doctor-api.hospital.com/register-order/add', {
    method: 'POST',
    headers: {
      'Content-Type': 'application/json',
      'token': getPatientToken()
    },
    body: JSON.stringify({
      doctorId,
      scheduleId,
      priority: 0
    })
  });
  const result = await response.json();
  return result.data; // 返回挂号单ID
}
```

### 5.2 病历查询集成

```javascript
// 1. 查询病历列表
const getMyMedicalRecords = async () => {
  const response = await fetch('http://doctor-api.hospital.com/medical-record/my-list', {
    headers: { 'token': getPatientToken() }
  });
  const result = await response.json();
  return result.data;
}

// 2. 查询病历详情
const getMedicalRecordDetail = async (recordId) => {
  const response = await fetch(`http://doctor-api.hospital.com/medical-record/patient-detail/${recordId}`, {
    headers: { 'token': getPatientToken() }
  });
  const result = await response.json();
  return result.data;
}
```

## 六、安全建议

### 6.1 权限验证
- 患者只能查看**自己的**病历，需在Service层添加权限校验
- 建议在 `getPatientMedicalRecordDetail` 方法中验证 `record.userId == patientId`

### 6.2 敏感数据脱敏
- 图片URL建议使用**临时授权链接**（如OSS签名URL，有效期1小时）
- 避免直接暴露永久存储路径

### 6.3 接口限流
- 对患者端接口添加限流（如每分钟最多10次请求）
- 防止恶意爬取病历数据

## 七、数据库表关联图

```
pmi_patient (患者表)
    ↓ id
register_order (挂号单)
    ↓ id = registerOrderId
medical_record (病历表)
    ↓ registerOrderId
medical_order (医嘱表)
    ↓ id = orderId
    ├─→ check_report (检查报告 - 含imageUrl/ctUrl)
    ├─→ check_order → lab_report (检验报告)
    └─→ prescription (处方表)
```

## 八、后续优化建议

1. **消息推送**：医生完成诊疗后，通过WebSocket/消息队列通知患者端更新病历
2. **病历状态同步**：增加 `medical_record.status` 字段（0草稿 1已完成），患者端仅展示已完成的病历

4. **数据缓存**：对历史病历数据进行Redis缓存，减少数据库查询
